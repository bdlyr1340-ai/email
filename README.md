# Atlas Digital Store — متجر رقمي مرن

نسخة أولى عملية لمتجر منتجات رقمية عربي/إنجليزي، مصممة للنشر على **Railway** وربطها بمستودع **GitHub**.

## الموجود حالياً

- واجهة متجر عربية وإنجليزية ومتجاوبة مع الهاتف.
- كتابة المنتج والخيار بالعربي فقط، ثم ترجمة الاسم والوصف إلى الإنجليزية تلقائياً بواسطة OpenAI API.
- منتجات بخيارات متعددة داخل نفس الصفحة: مشترك، شخصي، حساب كامل، كود، ملف، خدمة أو نوع مخصص.
- مخزون مشترك: الحساب الواحد ينشئ عدداً من الخانات تحدده أنت، مثلاً 5. بعد بيع الخانات الخمس يتوقف الحساب تلقائياً وينتقل النظام إلى الحساب التالي.
- مخزون شخصي: كل حساب أو كود يباع مرة واحدة فقط، ثم يرتبط دائماً بالطلب والمشتري.
- تسليم تلقائي، تسليم يدوي، أو تسليم الحساب أولاً ثم انتظار كود الدخول من الإدارة.
- رفع ملف TXT/CSV أو لصق الحسابات دفعة واحدة.
- الصيغ المقبولة لكل سطر:
  - `email|password|2FA`
  - `email|password`
  - `email`
  - ويمكن استخدام `:` أو `,` أو `;` أو Tab بدلاً من `|`.
  - JSON مثل: `{"email":"a@b.com","password":"123","two_factor":"SECRET"}`
- تشفير بيانات المخزون بواسطة AES-256-GCM قبل حفظها في PostgreSQL.
- إشعار بوت تليجرام عند إنشاء الطلب وعند الدفع، مع تنبيه خاص إذا كان الزبون ينتظر كوداً.
- حماية متعددة الطبقات: سؤال حسابي أو اختيار لون، Cloudflare Turnstile اختياري، rate limiting، منع كثرة الطلبات المعلقة، honeypot، وروابط طلبات سرية.
- دفع تجريبي جاهز للاختبار.
- موصل SuperQi/QiCard Sandbox وإنشاء رابط الدفع وWebhook موقّع.
- موصل Binance Pay Merchant قابل للتفعيل بعد إضافة بيانات حساب التاجر.

## لماذا المنتجات لا تختفي بعد التحديث؟

المنتجات والحسابات والطلبات محفوظة داخل **PostgreSQL** وليست داخل ملفات المشروع. عندما تدفع تحديثاً جديداً من GitHub، Railway يعيد بناء خدمة التطبيق فقط، وتبقى خدمة قاعدة البيانات كما هي. لا تحذف خدمة PostgreSQL أو الـVolume الخاص بها.

## تشغيل محلي

1. انسخ `.env.example` إلى `.env`.
2. ضع رابط PostgreSQL حقيقياً في `DATABASE_URL`.
3. ولّد مفاتيح الأمان بالأوامر الموجودة أدناه.
4. نفّذ:

```bash
npm install
npm run dev
```

المشروع ينشئ الجداول تلقائياً عند التشغيل.

## توليد متغيرات الأمان

```bash
# JWT_SECRET / CAPTCHA_SECRET / ABUSE_HASH_SECRET
openssl rand -base64 48

# INVENTORY_ENCRYPTION_KEY: يجب أن يكون 64 حرف hex بالضبط
openssl rand -hex 32
```

## النشر على GitHub وRailway

1. أنشئ مستودع GitHub جديداً وارفع محتويات هذا المجلد إليه.
2. في Railway اختر **New Project → Deploy from GitHub Repo** واختر المستودع.
3. داخل نفس مشروع Railway اختر **New → Database → PostgreSQL**.
4. في Variables لخدمة المتجر أضف:

```text
DATABASE_URL=${{Postgres.DATABASE_URL}}
```

5. أضف بقية المتغيرات من ملف `RAILWAY_VARIABLES.txt`.
6. من Settings → Networking أنشئ Public Domain.
7. ضع النطاق الناتج في `APP_URL` ثم أعد النشر.
8. افتح `/admin/login` وسجّل ببيانات `ADMIN_EMAIL` و`ADMIN_PASSWORD`.

## الترجمة التلقائية

ضع `OPENAI_API_KEY` وخلِّ `TRANSLATION_ENABLED=true`. عند إنشاء منتج أو خيار جديد، النظام يرسل النص العربي من السيرفر إلى OpenAI ويخزن النتيجة الإنجليزية في قاعدة البيانات. مفتاح OpenAI لا يظهر للمتصفح.

إذا لم تضع المفتاح، المتجر يعمل لكن النص الإنجليزي سيكون نفس العربي مؤقتاً.

## المخزون المشترك

عند اختيار `SHARED_SLOT`:

- تضيف الحساب الرئيسي مرة واحدة.
- تحدد `capacity` مثل 5.
- النظام ينشئ 5 خانات.
- كل عملية شراء تحجز خانة واحدة.
- بعد بيع الخامسة يتحول الحساب إلى غير نشط بدلاً من حذفه، حتى يبقى سجل المشترين محفوظاً.
- البيع التالي يأخذ خانة من الحساب الرئيسي التالي تلقائياً.

## انتظار كود الدخول

اختر `delivery_mode = WAIT_CODE` للخيار. بعد الدفع:

- يستلم الزبون البيانات المتوفرة مثل الإيميل والباسورد.
- تظهر له رسالة أنه ينتظر الكود.
- يصلك تنبيه في بوت الإدارة.
- تدخل إلى الطلب في لوحة الإدارة، تكتب الكود، فيظهر للزبون ويتحول الطلب إلى مكتمل.

## الدفع

### وضع التجربة

اترك:

```text
ENABLE_DEMO_PAYMENT=true
```

سيظهر SuperQi وBinance، والضغط على الدفع يؤكد الطلب تجريبياً لتستطيع اختبار المخزون والتسليم.

### SuperQi / QiCard

ابدأ ببيئة Sandbox ثم ضع بيانات Merchant Terminal. تحتاج عادةً إلى Terminal ID واسم مستخدم وكلمة مرور ومفتاح RSA عام للتحقق من Webhook. بعد نجاح الاختبار غيّر `SUPERQI_API_URL` إلى رابط الإنتاج الذي تزودك به QiCard.

### Binance Pay

يحتاج حساب **Binance Pay Merchant** وAPI Key/Secret ومفتاح Webhook العام. موصل الطلب موجود، لكن يجب اختبار إصدار الـMerchant API المتاح لحسابك قبل إغلاق وضع التجربة؛ قد تختلف صلاحيات وإصدارات الواجهة حسب حساب التاجر والمنطقة.

## ملاحظات إنتاج مهمة

- لا تضع أي Secret داخل GitHub؛ ضعها في Railway Variables فقط.
- فعّل Cloudflare Turnstile في الإنتاج.
- استخدم دومين HTTPS حقيقياً لروابط Webhook.
- لا تخزن الصور داخل نظام ملفات Railway المؤقت. استخدم رابط صورة ثابتاً أو فعّل Cloudinary/S3 لاحقاً.
- بع فقط المنتجات والحسابات التي تملك حق بيعها أو مشاركتها وفق شروط المنصة.
